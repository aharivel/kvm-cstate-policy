KVM_CAP_CSTATE_POLICY

Per-VM C-State Policy Enforcement — Proposal for KVM upstream

The Gap Today

KVM offers exactly two modes for guest idle management. There is no middle ground:

ModeHow it worksHost controlGuest autonomyVM exit cost
Default
cpu-pm=off		 Guest MWAIT/HLT causes VM exit → KVM intercepts → host decides C-state Full None Every halt
Delegated
cpu-pm=on		 Guest MWAIT executes directly on hardware, no VM exit None Full Zero
The problem: the client wants guest C-state autonomy (VNF owns power decisions) and a host-side kill switch (disable deep idle per-VM for debugging, no guest cooperation needed). Today these are mutually exclusive.

The Proposal

A new KVM capability that intercepts MWAIT, applies a per-VM policy, then executes the allowed C-state on behalf of the guest. The guest still decides what it wants — KVM enforces a ceiling.

Capability Definition

KVM_CAP_CSTATE_POLICY
  Architecture: x86
  Target:       VM
  Parameters:
    args[0] = max_cstate   // 0-6, or -1 for unrestricted
    args[1] = flags        // reserved, 0

Policy Values

max_cstateBehaviorUse case
-1 No interception — current cpu-pm=on behavior Full delegation
6 Guest drives C-states, KVM monitors but doesn't restrict Normal operation with observability
3 Allow up to C3 (33 µs exit), block C6 (133 µs exit) Gradual tuning
1 Guest requests C6 → KVM downgrades to C1. Kill switch. Debug / latency-sensitive
0 No idle at all (C0 only). Maximum power, minimum latency. Emergency

Internal Flow

Guest issues MWAIT with C-state hint │ ▼ KVM intercepts (VM exit on MWAIT) │ ▼ KVM reads kvm→max_cstate │ ├── requested ≤ max_cstate │ → execute native MWAIT with guest's hint │ └── requested > max_cstate → downgrade: MWAIT with max_cstate hint → or: skip MWAIT, return to guest (C0)

Design Questions

  1. Exit cost toggle. When max_cstate = -1, keep current cpu-pm=on behavior (no exits). When a policy is active, switch to interception. This means toggling VMCS execution controls at runtime — doable, needs care.
  2. MWAIT hint → C-state mapping. MWAIT sub-states don't map 1:1 to C-state names across CPU families. Policy should work on logical levels (C0/C1/C3/C6). KVM needs a translation table per CPU model.
  3. HLT vs MWAIT. Guests may use HLT (no sub-state hint). Policy must cover both: HLT with policy → KVM chooses C-state up to ceiling. MWAIT with policy → KVM caps the hint.
  4. Runtime toggle. Kill switch requires changing max_cstate on a live VM. The ioctl updates kvm→max_cstate atomically — takes effect on next halt/mwait.
  5. Interaction with halt_poll_ns. When interception is active, halt polling kicks in again. They become orthogonal: halt_poll_ns = how long to spin; max_cstate = deepest state allowed after.
  6. Per-vCPU vs per-VM. Start per-VM. Per-vCPU enables mixed policies (dataplane vCPUs in C0, control-plane in C6) but adds complexity.

The kill switch becomes a live virsh command or Nova API call that reaches KVM via ioctl. No guest cooperation. No redeployment.

Prior Art

  • KVM_CAP_HALT_POLL — per-VM halt polling override. Same design pattern: VM capability, ioctl, per-VM value. Merged 2020.
  • Xen ACPI idle driver — per-domain C-state policy via ACPI integration. KVM lacks this entirely.
  • cpuidle governors (menu, TEO, haltpoll) — host-side C-state selection. This proposal moves that to a per-VM policy.
  • AWS Nitro / Azure Boost — cloud hypervisors likely have internal per-VM idle policies (proprietary, not upstream).

Impact

If merged: first upstream mechanism giving cloud operators per-VM, live, host-enforced C-state control without guest cooperation. Fills the gap between "host controls everything" and "guest controls everything."

For NFV: VNF vendor sets preferred C-states via guest cpuidle. Platform operator sets the ceiling via flavor/policy. Debug sessions override to C0/C1 with a single virsh command. No redeployment. No guest agent. No SSH.
